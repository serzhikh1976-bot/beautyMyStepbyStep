# Beauty Platform — Мультибот-платформа для бьюти-мастеров

Мультибот-платформа для агрегации бьюти-мастеров на базе Telegram. Один Node.js-сервер обслуживает неограниченное количество Telegram-ботов. Каждый бот = один город.

## Стек технологий

- **Runtime:** Node.js 18+, TypeScript
- **HTTP-сервер:** Fastify (+ `@fastify/cookie`, `@fastify/formbody` для веб-панели)
- **Telegram-фреймворк:** Ultra Telegram Framework (UTF)
- **База данных:** Supabase (PostgreSQL)
- **Деплой:** ngrok (разработка) → Railway → VPS

---

## Архитектура

### Мультибот (Паттерн Singleton)

```
Telegram → POST /webhook/:uuid → Fastify → Map<uuid, TelegramBot> → handleUpdate()
```

- Единый URL `/webhook/:uuid` для всех ботов
- При первом запросе токен достаётся из БД по `uuid` (поле `bots.number`)
- Инстанс бота кешируется в `Map` — повторные запросы не ходят в БД
- Токен **никогда не передаётся** по сети (только `uuid` в URL)
- Инвалидация кеша через `invalidateBot(uuid)` при изменении токена

### Изоляция данных

Все таблицы содержат `bot_id`. Данные разных городов физически не пересекаются.

## База данных (Supabase)

### Таблицы

| Таблица | Назначение |
|---|---|
| `bots` | Реестр ботов (город, токен, менеджер) |
| `districts` | Районы города |
| `sub_districts` | Подрайоны |
| `users` | Пользователи с ролью (client/master) |
| `services` | Глобальный справочник услуг |
| `bot_services` | Услуги включённые в конкретном боте |
| `masters_profiles` | Анкеты мастеров |
| `master_services` | Услуги мастера (many-to-many) |
| `active_chats` | Активные анонимные чаты. Плюс служебные колонки `last_master_button_msg_id` / `last_client_button_msg_id` — id последнего сообщения с кнопками на каждой стороне (см. «Самоочищающиеся кнопки») |
| `chat_messages` | Маппинг `message_id` для Reply-туннеля (`chat_id` → `message_id`, привязка к боту — через `active_chats.bot_id`) |

| `blocked_clients` | Баны: какой мастер (`master_id`) заблокировал какого клиента (`client_id`) в каком боте (`bot_id`) |
| `orders` | Заявки клиентов (биржа, следующий этап) |
| `order_responses` | Отклики мастеров на заявки |
| `broadcast_messages` | Лог веерной рассылки |

### Автоочистка логов

Через Supabase Extensions → **pg_cron**:

```sql
select cron.schedule(
  'clean-chat-logs',
  '0 3 * * *',
  $$delete from chat_message_log where created_at < now() - interval '14 days'$$
);
```

## Функционал

### Роли

При первом запуске `/start` пользователь выбирает роль: **Клиент** или **Мастер**. Роль сохраняется в таблице `users` и сессии.

### Мастер

**Регистрация** (WizardScene, 5 шагов):
1. Имя
2. Услуги — мультиселект из глобального справочника
3. Район → Подрайон (если есть)
4. Минимальная цена
5. Фото портфолио (до 5 штук, `file_id` от Telegram)

После регистрации выдаётся **триал на 30 дней** (`trial_expires_at`).

**Профиль** (`👤 Мой профиль`):
- Альбом из всех фото портфолио
- Статус: Активен / На паузе
- Кнопки управления: `⏸ Пауза`, `✏️ Фото`, `💰 Цена`, `📍 Район`, `🔧 Услуги`

**Редактирование** (отдельные WizardScene для каждого поля):
- Цена — вводится числом
- Район/Подрайон — инлайн-кнопки
- Услуги — мультиселект с тоглами ✅/☐
- Фото — полная замена портфолио (до 5 фото, `/done` или автоматически при 5)

### Клиент

**Умный поиск** (фильтрация на каждом шаге):
1. Услуга — только те где есть активные мастера
2. Район — только те где есть мастера с выбранной услугой
3. Подрайон — только те где есть мастера (если существуют)

Результат — список мастеров инлайн-кнопками:
```
[Мария · Маникюр, Педикюр · от 300 грн]
[Анна · Стрижка · от 200 грн]
```

Нажатие → полная карточка мастера (альбом фото + все услуги + район + цена).

### Анонимный чат

```
Клиент → 💬 Написать мастеру → туннель → Мастер
```

- Проверка: у клиента может быть только **1 активный чат** — причём проверяется **сразу** при нажатии «🔍 Найти мастера» (не в конце визарда поиска)
- Проверка: мастер должен быть `is_active = true`
- Проверка: клиент не должен быть в `blocked_clients` у этого мастера
- Мастеру приходит уведомление + `message_id` сохраняется для маппинга Reply
- **Клиент пишет** → мастер получает `💬 Имя: текст`
- **Мастер отвечает:**
  - 1 активный чат → любое сообщение (Reply не обязателен)
  - Несколько чатов → обязательно через Reply на нужное сообщение клиента
  - Reply на сообщение из **уже завершённого** чата → явное предупреждение «Этот диалог уже завершён», сообщение никуда не уходит (раньше молча улетало в единственный оставшийся активный чат — это было исправлено)
- Поддержка **фото** в обе стороны
- **Завершение:** кнопка `❌ Завершить диалог`, с подтверждением в два шага (защита от случайного нажатия), у обеих сторон отображается имя собеседника
- **Бан:** у мастера рядом с «Завершить» — кнопка `🚫 Забанить` (тоже с подтверждением). Забаненный клиент видит мастера в поиске, но не может написать
- **Лог переписки** в `chat_message_log` (текст + `photo_ids[]`, хранится 14 дней)

#### Изоляция между городами

Каждый бот = отдельный город, но все города используют одну и ту же БД и один процесс. Изоляция строго на уровне запросов — везде фильтр по `bot_id`. Отдельно стоит отметить: `chat_messages.message_id` — это Telegram `message_id`, который уникален **только в рамках одного бота**, а не глобально (у каждого бота своя независимая нумерация). Поэтому Reply-роутинг всегда джойнит через `active_chats` и фильтрует `active_chats.bot_id`, а не полагается на голый `message_id` — иначе при совпадении `message_id` у двух разных ботов сообщение могло бы улететь в чужой город.

#### Самоочищающиеся кнопки

Inline-кнопки в Telegram не «протухают» сами — старое сообщение с кнопкой в истории переписки остаётся кликабельным вечно. Это создавало реальные баги: клиент мог случайно нажать старую кнопку «Написать мастеру» повторно, мастер — случайно повторно забанить уже разбаненного клиента, кликнув на сообщение недельной давности.

Решение — функция `sendTracked()` в `bot-manager.ts`: перед отправкой нового сообщения с кнопками она смотрит в `active_chats.last_master_button_msg_id` / `last_client_button_msg_id`, стирает кнопки с предыдущего сообщения на этой стороне (`editMessageReplyMarkup` с пустой клавиатурой) и запоминает id нового. Работает независимо для каждой стороны чата. Плюс отдельный хелпер `clearButtons()` — для одноразовых действий вне чат-туннеля (выбор роли на `/start`, шаги визарда поиска: услуга → район → подрайон → выбор мастера).

Правило простое: **одноразовое действие с необратимым эффектом → кнопка исчезает после клика. Постоянное меню для навигации/редактирования (список результатов поиска, меню «Мой профиль») → остаётся кликабельным.**

## Администрирование

Помимо обычных city-ботов есть два независимых способа управления всей платформой сразу — оба доступны только `SUPER_ADMIN_ID`, оба не привязаны к таблице `bots` (не являются городом), конфигурируются только через `.env`.

### Admin-бот (Telegram)

Отдельный Telegram-бот (`ADMIN_BOT_TOKEN` / `ADMIN_BOT_UUID`), видит сводку по **всем городам сразу**:

- `/start` → дашборд: сколько городов активно, сколько мастеров, сколько открытых чатов
- Клик по городу → детали: статус, список мастеров, число открытых чатов
- «Все активные чаты» → список диалогов по всем городам с ID клиента/мастера

Middleware в начале `admin-bot.ts` отсекает всех, кроме `SUPER_ADMIN_ID`, — остальным бот просто не отвечает.

### Веб-панель администратора (`/admin`)

Тот же Fastify-процесс, что обслуживает вебхуки, отдаёт ещё и HTML-панель. Вход по паролю (`ADMIN_PANEL_PASSWORD`), сессия — подписанная cookie (`ADMIN_PANEL_SESSION_SECRET`, через `@fastify/cookie`), без БД и без in-memory хранилища — переживает рестарт сервера.

| Раздел | Что можно |
|---|---|
| `/admin` | Все мастера по всем городам: фильтр по городу/статусу, поиск по имени, сортировка кликом по заголовку |
| `/admin/chats` | Список всех чатов (активных и завершённых), фильтры по городу/статусу/Client ID/Master ID |
| `/admin/chats/:id` | Полная переписка в виде диалоговых пузырей, с фото (превью 160×160, клик открывает оригинал) — для разбора жалоб |
| `/admin/clients` | Уникальные клиенты по городам, число чатов, последняя активность |
| `/admin/bans` | Все баны по всем городам, кнопка «Разбанить» (также доступна прямо на странице переписки, если чат забанен) |
| `/admin/bots` | Список городов, вкл/выкл одной кнопкой, форма «Добавить город» (токен + название → вебхук регистрируется у Telegram автоматически) |
| `/admin/services` | Глобальный каталог услуг + вкл/выкл для выбранного города, создание новой услуги |
| `/admin/districts` | Районы и подрайоны по городам, добавление/удаление |

Технические детали:
- Фото из чатов подгружаются через прокси-эндпоинт `/admin/photo/:botId/:fileId` — сервер сам ходит в Telegram API по токену нужного city-бота и стримит байты, не светя токен в HTML.
- Сортировка мастеров по городу делается **в JS после запроса**, а не через `.order(..., { referencedTable })` — PostgREST не умеет сортировать внешние строки запроса по полю из связанной таблицы для связи «многие-к-одному» (эта опция сортирует только вложенные строки внутри embedded-relation, не сам верхнеуровневый список).
- Все embedded-джойны вида `active_chats.select('..., masters_profiles(name)')` заменены на два последовательных запроса — прямой FK между `active_chats.master_id` и `masters_profiles.master_id` не был явно оформлен в схеме, из-за чего PostgREST мог вернуть ошибку, которая раньше нигде не логировалась и приводила к тихим сбоям проверок (например, «у вас уже активный чат» не срабатывало).
- При выключении бота (`is_active=false`) вызывается `invalidateBot()` — кэшированный в памяти инстанс сбрасывается сразу, а не после рестарта процесса.

## Установка и запуск

### Требования

- Node.js 18+
- Аккаунт Supabase
- Токен(ы) ботов от @BotFather
- ngrok (для разработки)

### 1. Клонирование и установка зависимостей

```bash
git clone <repo>
cd beauty-platform
npm install
```

### 2. Настройка Supabase

1. Создайте проект на [supabase.com](https://supabase.com)
2. **SQL Editor** → создайте таблицы `bots`, `districts`, `sub_districts`, `users`, `services`, `bot_services`, `masters_profiles`, `master_services`, `active_chats`, `chat_messages`, `chat_message_log`, `orders`, `order_responses`, `broadcast_messages` (полного дампа схемы пока нет в репозитории — см. Roadmap) → затем накатите миграции из `supabase/`: `migration_blocked_clients.sql`, `migration_button_tracking.sql`
3. **Extensions** → включите `pg_cron`
4. Выполните SQL для автоочистки логов (см. раздел "База данных")
5. **Project Settings → API** → скопируйте `Project URL` и ключ `service_role`

### 3. Переменные окружения

```bash
cp .env.example .env
```

Заполните `.env`:

```env
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-service-role-key
PORT=3000
PUBLIC_BASE_URL=https://xxxx.ngrok-free.app
SUPER_ADMIN_ID=ваш_telegram_id

# Admin-бот в Telegram (сводка по всем городам, см. раздел «Администрирование»)
ADMIN_BOT_TOKEN=токен_от_botfather
ADMIN_BOT_UUID=случайный-uuid-для-урла-вебхука

# Веб-панель администратора (/admin)
ADMIN_PANEL_PASSWORD=придумайте_надёжный_пароль
ADMIN_PANEL_SESSION_SECRET=случайная-длинная-строка-для-подписи-cookie
```

### 4. Добавление бота, услуг и районов

Проще всего — через веб-панель (`/admin/bots`, `/admin/services`, `/admin/districts`), она сама зарегистрирует вебхук у Telegram. Но можно и напрямую SQL, если панель ещё не поднята:

```sql
insert into bots (number, city_name, token, is_active, manager_telegram_id)
values (gen_random_uuid(), 'Киев', '123456:ABC-TOKEN', true, 123456789);
```

Заполните справочник услуг:

```sql
insert into services (name) values
  ('Маникюр'), ('Педикюр'), ('Стрижка'),
  ('Окрашивание'), ('Массаж'), ('Брови');
```

Включите услуги для бота:

```sql
insert into bot_services (bot_id, service_id, is_enabled)
select b.id, s.id, true from bots b, services s
where b.city_name = 'Киев';
```

Добавьте районы:

```sql
insert into districts (bot_id, name)
select id, 'Центральный' from bots where city_name = 'Киев';

-- Подрайоны (опционально):
insert into sub_districts (district_id, name)
values (<district_id>, 'Старый город');
```

## Известные баги UTF и обходные пути

В процессе разработки обнаружено несколько багов в `ultra-telegram-framework`:

### 1. `ctx.from` возвращает неверный ID в action-хендлерах

**Проблема:** `ctx.from.id` в `bot.action()` возвращает ID бота, а не пользователя.

**Причина:** `get from()` читает из `ctx.message.from` (автора сообщения с кнопками = бот), а не из `callbackQuery.from`.

**Решение:** Всегда использовать `ctx.callbackQuery!.from.id` в action-хендлерах.

### 2. Неверный ключ сессии при callback_query

**Проблема:** Стандартный `getSessionKey` тоже использует `ctx.from`, из-за чего сессии клиента и мастера в одном боте перемешиваются.

**Решение:** Кастомный `getSessionKey`:

```typescript
getSessionKey: (ctx) => {
  const userId = ctx.callbackQuery?.from.id ??
    (ctx.message && 'from' in ctx.message ? ctx.message.from?.id : undefined);
  const chatId = ctx.chatId;
  if (!userId || !chatId) return undefined;
  return `${record.id}:${chatId}:${userId}`; // включаем botId для изоляции
}
```

### 3. `scene.enter()` не запускает step 0 немедленно

**Проблема:** `ctx.scene.enter('scene_name')` только записывает сцену в сессию. Step 0 запускается на **следующем** апдейте.

**Решение:** Показывать вопрос/клавиатуру **до** `scene.enter()` в action/match-хендлере. Step 0 обрабатывает только ответ пользователя.

### 4. `scene.next()` немедленно триггерит следующий шаг

**Проблема:** После `ctx.scene.next()` следующий шаг вызывается сразу с текущим ctx (callback_query), что ломает шаги ожидающие текст.

**Решение:** Защитные проверки в начале каждого шага:

```typescript
// Шаг ожидает текст:
async (ctx) => {
  if (!ctx.text) return;
  // ...
}

// Шаг ожидает callback:
async (ctx) => {
  if (!ctx.callbackQuery) return;
  // ...
}
```

### 5. `scene.state` сбрасывается при `scene.enter()`

**Проблема:** Данные записанные в `ctx.scene.state` ДО `scene.enter()` теряются.

**Решение:** Устанавливать `scene.state` ПОСЛЕ `scene.enter()`:

```typescript
ctx.scene.enter('my_scene');
ctx.scene.state.data = value; // ✅ после enter
```
